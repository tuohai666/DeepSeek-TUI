# 01 · 项目概览

> 这一节回答三个问题：**它是什么 / 它为什么这样写 / 它如何与同类项目区分开**。
> 读完之后，你脑子里应该已经有了一张"主干图"，再读后续章节才不会迷路。

---

## 1. 它是什么

**DeepSeek-TUI** 是一个跑在终端里的 **LLM 编码 Agent**：

- 你输入"帮我把这个函数拆开" →
- 它在本地工作区里 **读文件、跑 shell、做 grep、写补丁、跑测试** →
- 把每一步通过 ratatui 流式渲染给你看 →
- 命中危险动作时弹审批 →
- 最后把结果落地到磁盘并能 `/restore` 回滚。

它本质上是一个**多组件协作的运行时**，不是 OpenAI SDK 的 thin wrapper。它处理的核心难题是：

| 难题 | 它的解法 |
|---|---|
| LLM 是阻塞的，UI 必须流畅 | 用 `tokio::mpsc` 把 *引擎* 和 *UI* 拆成两个 task |
| 模型会要求执行危险命令 | 三档模式（Plan / Agent / YOLO）+ 工具级别审批 |
| 一次任务可能跑几十轮、上下文爆掉 | 上下文压缩 + cycle 切片 + 容量守门员 |
| 长跑任务进程崩溃丢进度 | checkpoint + side-git snapshot + 任务管理器 |
| 单 LLM 一次推不完复杂任务 | 子代理 (`agent_spawn`) + RLM (Python REPL 里调子 LLM) |
| 不同团队用不同推理后端 | Provider 抽象（DeepSeek / NVIDIA NIM / Fireworks / SGLang / vLLM / Ollama / OpenRouter / OpenAI） |
| 编辑后立刻知道是否编译过 | LSP 集成，编辑工具完成后自动收 diagnostics |

**关键事实**：这个项目把"做一个能上线的 Agent"所需的全部边界条件都写进代码了，不是 demo。

## 2. 它的物理结构

仓库根是个 **Cargo workspace**，看 `@D:\1os\DeepSeek-TUI-tuohai666\Cargo.toml:1-19`：

```toml
[workspace]
members = [
    "crates/agent",
    "crates/app-server",
    "crates/cli",
    "crates/config",
    "crates/core",
    "crates/execpolicy",
    "crates/hooks",
    "crates/mcp",
    "crates/protocol",
    "crates/secrets",
    "crates/state",
    "crates/tools",
    "crates/tui",
    "crates/tui-core",
]
default-members = ["crates/cli", "crates/app-server", "crates/tui"]
```

14 个 crate，**别被吓到**，实际生产真正在跑的只有两个：

- **`crates/cli`** —— 编出 `deepseek` 派发器二进制。极薄，只做参数解析和环境变量整理。
- **`crates/tui`** —— 编出 `deepseek-tui` 主二进制。**项目 90% 的实质代码都在这里**：引擎、工具、客户端、UI、会话、MCP、LSP、子代理…

其余 12 个 crate 大多是 **正在被切出去的薄层**（见 `@D:\1os\DeepSeek-TUI-tuohai666\docs\ARCHITECTURE.md:5-11` 的"边界注释"）：
- `crates/tools` 提供 `ToolSpec` 共享 trait
- `crates/agent` 提供 ModelRegistry
- `crates/secrets` 提供 keyring 抽象
- `crates/state` 提供 SQLite 持久化
- `crates/execpolicy` 提供沙箱策略
- 其他几个还在演进，**学习时优先看 `crates/tui` 内部**就够了。

> **学习窍门**：把 `crates/cli/src/lib.rs` 当成"门厅"，`crates/tui/src/` 当成"主楼"。门厅只是引你进门，主楼里才有所有家具。

## 3. 两段二进制：为什么不合成一个

直觉上，一个 CLI 工具应该编出一个 binary。这里却编出 **两个**：`deepseek` 和 `deepseek-tui`。

来源：`@D:\1os\DeepSeek-TUI-tuohai666\crates\cli\src\lib.rs:1407-1441` 的 `locate_sibling_tui_binary`。

**为什么这样拆？**

1. **派发器极轻量** —— `cargo install deepseek-tui-cli` 不必拉 `ratatui` / `crossterm` 这一大坨终端依赖，便于做 `/etc/deepseek` 类的运维路径。
2. **MCP 服务器模式 / app-server 模式 / acp 模式** 不需要 TUI 渲染，但仍可复用 `deepseek` 命令入口。
3. **崩溃隔离** —— TUI panic 时派发器已经退出，避免把交互终端长期占住。

派发器干的事其实很小：
1. `clap` 解析顶层指令；
2. 把 `--api-key`、`--model`、`--base-url` 等用户给的覆盖项**变成环境变量**；
3. `Command::new(deepseek-tui).args(passthrough).status()` 调起伴生二进制；
4. 转发退出码。

第 03 节会逐行带读这段。

## 4. 关键设计原则（这是本节最该带走的）

### 4.1 *Streaming-first*：所有路径都假设能流式

整个数据栈每一层都不缓存"完整的响应"再交给下游：

- `handle_chat_completion_stream` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\client\chat.rs:137`) 用 `reqwest` + SSE，逐 `data:` 行解析为 `StreamEvent`。
- `turn_loop` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:336-684`) 一个 `tokio::select` 边收 chunk 边发 `Event::MessageDelta`。
- `run_event_loop` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tui\ui.rs:686`) 收到 delta 立刻 push 到 `streaming_state`，下一个绘帧周期就把字打到屏幕。

**得到的好处**：取消请求时丢掉的工作很少；用户看到字在打出来，体感等待 = 0。
**付出的代价**：每一层都得处理"半成品状态"，比如"工具调用 JSON 还没收完整"——见 `tool_state.input_buffer` 的累积逻辑。

### 4.2 *Channel-driven*：把组件解耦成"消息流"

```
UI ──Op──▶ Engine ──Event──▶ UI
              │
              ├──▶ ToolRegistry::execute_full_with_context
              ├──▶ DeepSeekClient::create_message_stream
              └──▶ SubAgentRuntime::spawn_background
```

`Op` 和 `Event` 都是 enum，分别定义在 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\ops.rs:14-98` 和 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\events.rs:27-258`。

**关键不变量**：UI 永远不直接读模型响应，引擎也永远不直接画屏幕。它们只通过 `mpsc::Sender/Receiver` 通信。

这套范式让你可以：
- 把 `Engine` 的同一份 `Op` 输入给 HTTP runtime API（`runtime_api.rs`）和 ACP server（`acp_server.rs`）——它们用同一个引擎，只是源/汇不一样。
- 单元测试 Engine 时，只需要 mock `mpsc::Receiver<Op>` 和 `mpsc::Sender<Event>`。

### 4.3 *Tool-as-typed-protocol*：模型只能通过工具协议影响世界

模型的全部"行动力"都来自 `ToolSpec`：

```rust
// @D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\spec.rs:548-571
#[async_trait]
pub trait ToolSpec: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    fn capabilities(&self) -> Vec<ToolCapability>;
    fn approval_requirement(&self) -> ApprovalRequirement { /* derived from caps */ }
    /* … */
}
```

每个工具自描述：名字、文档、JSON Schema、能力列表（`ReadOnly` / `WritesFiles` / `ExecutesCode`）。引擎根据能力**自动**决定是否要走审批闸门。

为什么这样设计？因为 LLM 给你的是一个 `tool_use` block 加 JSON 参数，引擎必须能：
1. 校验参数（schema）；
2. 决定要不要批准（capability → approval）；
3. 决定能不能并行（write 类不能并行）；
4. 决定是否触发 LSP 后处理（写文件后挂 diagnostics）。

只有当所有这些信息都长在工具自身上时，引擎才能写成一段统一的调度循环——这正是 `turn_loop` 里"plans → outcomes"的两阶段做的事。

### 4.4 *Stable-only Rust*：不用 nightly feature

来自 `@D:\1os\DeepSeek-TUI-tuohai666\AGENTS.md` 项目规则：必须能在 stable Rust 上编。
所以你看代码里大量用 `&& let Some(_) = …`（**Rust 1.88 才稳定的 let_chains**），但 *永远不会* 看到 `#![feature(...)]`。

学习意义：你接下来做改动时也不能写 `if let` guard、不能上 `try_blocks`，会被 CI 直接刷掉。

### 4.5 *Local-first*：所有持久化都在 `~/.deepseek/`

看 `@D:\1os\DeepSeek-TUI-tuohai666\docs\ARCHITECTURE.md:288-300`：

```
~/.deepseek/config.toml          # 配置 + API Key
~/.deepseek/sessions/            # 会话历史
~/.deepseek/sessions/checkpoints # 崩溃 checkpoint + 离线队列
~/.deepseek/snapshots/           # side-git 工作区快照
~/.deepseek/tasks/               # 后台任务记录
~/.deepseek/audit.log            # 审计日志
```

没有云后端，没有数据库服务，**进程关掉再开就能恢复一切**——这是它能在断网时仍然让你排队、再连时自动 flush 的根本原因。

## 5. 学习中你需要建立的"概念字典"

后面九篇会反复出现下面这些词，先在脑里挂上简译：

| 概念 | 在哪 | 一句话解释 |
|---|---|---|
| **Op** | `core/ops.rs:14` | UI → Engine 的命令，如 `SendMessage`、`CompactContext` |
| **Event** | `core/events.rs:27` | Engine → UI 的消息，如 `MessageDelta`、`ToolCallStarted` |
| **Turn** | `core/turn.rs:23` | "一次用户消息 + 模型响应"为一轮 |
| **Step** | `TurnContext.step` | 一轮里第 N 次模型调用（每次工具调用后会再喂回模型） |
| **Mode** | `tui/app.rs:89` | Plan（只读） / Agent（要审批） / YOLO（自动批） |
| **ToolSpec** | `tools/spec.rs:548` | 一切工具的抽象 |
| **ToolContext** | `tools/spec.rs:71` | 工具执行时拿到的环境（workspace、shell mgr、network policy） |
| **ContentBlock** | `models.rs` | 一条 message 的内容块：`Text` / `Thinking` / `ToolUse` / `ToolResult` |
| **Sub-agent** | `tools/subagent/mod.rs` | 子任务跑独立 LLM 会话 |
| **RLM** | `tools/rlm.rs` + `repl/` | 把长输入丢进 Python REPL，让一个子 LLM 在里面用 `llm_query()` 来切片处理 |
| **Cycle** | `cycle_manager.rs` | 上下文压力达到阈值时整轮翻篇，留下 briefing |
| **Snapshot** | `snapshot/` | 用一个 *side git repo* 快照工作区，与你的 `.git` 隔离 |
| **MCP** | `mcp.rs` | Model Context Protocol，外部工具服务发现协议 |
| **Skill** | `skills/` | Markdown + 脚本组成的可装载"行为模板" |
