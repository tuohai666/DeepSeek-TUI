# 05 · 工具体系：从 ToolSpec 到子代理

> 第 04 节的最后两步把球交给了"工具调度"。这一节我们站在调度的下游，看清楚**工具是如何自描述、被注册、被审批、被执行**，以及"子代理"和"RLM"为什么本质上**也只是两个特殊的工具**。

## 1. 工具体系的"三层抽象"

```
┌───────────── 模型视角 ─────────────┐
│  tool_use { name, input } JSON     │
└──────────────┬─────────────────────┘
               │ LLM 输出
┌──────────────▼─────────────────────┐
│ ToolRegistry::execute_full_with_   │  ← 调度层
│ context(name, input, ctx)          │
└──────────────┬─────────────────────┘
               │ resolve + invoke
┌──────────────▼─────────────────────┐
│ Arc<dyn ToolSpec>::execute(...)    │  ← 抽象层
│ ┌──────────────────────────────┐   │
│ │ ReadFileTool                 │   │  ← 实现层（25+ 个）
│ │ EditFileTool                 │   │
│ │ ExecShellTool                │   │
│ │ AgentSpawnTool               │   │
│ │ RlmTool                      │   │
│ │ … 还有 MCP 适配器 …         │   │
│ └──────────────────────────────┘   │
└────────────────────────────────────┘
```

**关键观察**：从 LLM 角度看，`agent_spawn` 和 `read_file` 没有区别——都是 JSON in / JSON out 的工具。子代理的复杂度被完全藏在 `AgentSpawnTool::execute` 里。

## 2. `ToolSpec` trait：工具的合同

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\spec.rs:548-602`：

```rust
#[async_trait]
pub trait ToolSpec: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    fn capabilities(&self) -> Vec<ToolCapability>;

    fn approval_requirement(&self) -> ApprovalRequirement {
        let caps = self.capabilities();
        if caps.contains(&ToolCapability::ExecutesCode) {
            ApprovalRequirement::Required
        } else if caps.contains(&ToolCapability::WritesFiles) {
            ApprovalRequirement::Suggest
        } else {
            ApprovalRequirement::Auto
        }
    }

    fn is_read_only(&self) -> bool { /* … */ }
    fn supports_parallel(&self) -> bool { false }
    fn defer_loading(&self) -> bool { false }

    async fn execute(&self, input: Value, context: &ToolContext) -> Result<ToolResult, ToolError>;
}
```

每个方法都对应"调度层要回答的某个问题"。看下表你就明白这套设计的**经济性**：

| 方法 | 调度层要回答的问题 |
|---|---|
| `name()` | 模型说要调 `read_file`，是哪个工具？ |
| `description()` + `input_schema()` | 我要把这个工具序列化进 `tools` 数组发给模型。 |
| `capabilities()` | 这个工具会改文件 / 执行代码 / 走网络吗？ |
| `approval_requirement()` | 要不要弹审批？ |
| `is_read_only()` | 能并行吗？要不要拿写锁？ |
| `supports_parallel()` | 即使是只读，作者本身允许并行吗？ |
| `defer_loading()` | 把它放进 catalog，但**不**显示给模型，等模型主动 `tool_search` 才暴露。 |
| `execute(input, ctx)` | 真正的副作用 |

**为什么 capabilities 是一个 Vec 而不是一个 enum？** 因为一个工具可能同时是 `WritesFiles + ChecksRepo`、`ExecutesCode + Sandboxable`。组合多种能力比强行划分一个固定枚举更灵活。

## 3. `ToolContext`：工具拿到的世界

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\spec.rs:71-138`，挑核心字段：

```rust
pub struct ToolContext {
    pub workspace: PathBuf,                                  // 工作区根
    pub shell_manager: SharedShellManager,                   // 共享 shell
    pub trust_mode: bool,                                    // 是否绕路径检查
    pub auto_approve: bool,                                  // YOLO 模式
    pub features: Features,                                  // 特性开关
    pub state_namespace: String,                             // 状态命名空间
    pub trusted_external_paths: Vec<PathBuf>,                // 用户加白的外部路径
    pub network_policy: Option<NetworkPolicyDecider>,        // 域名/端口策略
    pub runtime: RuntimeToolServices,                        // 任务管理器、自动化等
    pub cancel_token: Option<CancellationToken>,             // 取消信号
    pub sandbox_backend: Option<Arc<dyn SandboxBackend>>,    // 沙箱
    pub memory_path: Option<PathBuf>,                        // 用户 memory 文件
    pub lsp_manager: Option<Arc<LspManager>>,                // LSP 后处理
    pub large_output_router: Option<…>,                      // 大输出路由
    pub workshop_vars: Option<Arc<Mutex<WorkshopVariables>>>,// workshop 替身
    /* … */
}
```

**学习要点**：`ToolContext` 是工具与"全局状态"的**唯一接口**。工具不能 `static` 全局变量，不能直接访问 `Engine`。这种"工具只会拿到 context 这一只袋子"的设计有两个好处：
1. **测试容易** —— 给一个 `ToolContext::new(tempdir())` 就能测；
2. **多实例安全** —— 子代理跑独立 `ToolContext`，不会相互踩到（每个子代理一个新的 `state_namespace`）。

### 3.1 `resolve_path` —— 沙箱的第一道闸

`ToolContext::resolve_path` 把模型给的字符串路径转成绝对路径，并**强制锚定在 workspace 内**（除非 `trust_mode = true` 或路径在 `trusted_external_paths`）。来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\spec.rs:642-723` 的测试：

```rust
ctx.resolve_path("../escape.txt")          // → Err(PathEscape)
ctx.resolve_path("new/../safe.txt")        // → Ok(workspace/safe.txt)
ctx.resolve_path("/etc/passwd")            // → Err(PathEscape) （非 trust_mode）
```

这是**"模型不能想读什么就读什么"**的硬性约束，每个文件类工具都先调它。

### 4.2 序列化：`to_api_tools` 与稳定排序

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\registry.rs:217-238`：

```rust
fn build_api_tools(&self) -> Vec<Tool> {
    let mut tools: Vec<&Arc<dyn ToolSpec>> = self.tools.values().collect();
    tools.sort_by(|a, b| a.name().cmp(b.name()));
    tools.into_iter().map(|tool| Tool {
        name: tool.name().to_string(),
        description: tool.description().to_string(),
        input_schema: { let mut s = tool.input_schema(); schema_sanitize::sanitize(&mut s); s },
        defer_loading: Some(tool.defer_loading()),
        /* … */
    }).collect()
}
```

注意三件事：
1. **`sort_by(name)` 强制稳定顺序** —— 保证 KV cache 命中（这点在 04 节也提过）。
2. **`schema_sanitize::sanitize` 清洗 schema** —— DeepSeek API 拒绝某些 JSON Schema 关键字（如 `default`、`example`），统一脱敏。
3. **`OnceLock + clone()` 缓存结果** —— 同一会话里这个数组永远不变，每次请求复用同一份。

### 4.3 执行：`execute_full_with_context`

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\registry.rs:124-199`，简化骨架：

```rust
pub async fn execute_full_with_context(
    &self,
    name: &str,
    input: Value,
    ctx: Option<ToolContext>,
) -> Result<ToolResult, ToolError> {
    let resolved_name = self.resolve(name).unwrap_or(name);
    let Some(tool) = self.tools.get(resolved_name) else {
        return Err(ToolError::not_available(format!("Unknown tool: {name}")));
    };
    let ctx = ctx.unwrap_or_else(|| self.context.clone());

    /* 1. 跑 before-execute hook（model 检查、特性开关） */
    /* 2. 调 tool.execute(input, &ctx).await */
    let result = tool.execute(input, &ctx).await?;
    /* 3. 大输出路由（>阈值 token 时走 workshop synthesis） */
    let routed = ctx.large_output_router.as_ref()
        .map(|router| router.maybe_route(&name, &result))
        .unwrap_or(result);
    Ok(routed)
}
```

注意第 1 步的 `resolve(name)`：**模型经常说错工具名**，如把 `read_file` 写成 `ReadFile`、`read-file`、`readfile`。`resolve` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\registry.rs:341-383`) 用一条五步阶梯做模糊匹配：

1. 大小写折叠精确
2. 横线/空格 → 下划线
3. CamelCase → snake_case
4. 剥 `_tool` / `tool` 后缀
5. 前后缀模糊

> **学习要点**：在 LLM 应用里，**"宽容一切错误，但内部只用规范名"** 是黄金原则。对错误的工具名宁可静默纠正也不要让模型卡住——纠正一次的成本远低于让模型重试。

## 5. 审批闸门：能力 → 决定

引擎在 turn_loop 里把能力转化为 *"这次要不要弹对话框"*：

```
ApprovalRequirement::Auto      ──▶ 直接执行
ApprovalRequirement::Suggest   ──▶ 弹审批，但默认 Yes（写文件类）
ApprovalRequirement::Required  ──▶ 弹审批，必须显式 Yes（执行命令类）
```

在 *YOLO 模式* 下：
- `Suggest` 自动 Yes（不弹）
- `Required` 仍然弹（保留底线，否则太危险）

在 *YOLO + auto_approve = true* 下：
- 全部自动 Yes（这是 `--dangerously-bypass-approvals` 的语义）

实现：`turn_loop.rs` 的 `decide_approval_strategy` 在 plans 阶段查工具能力，决定要不要发 `Event::ApprovalRequired`，UI 弹对话框，用户决定后通过 `rx_approval` 回到引擎。

> **学习要点**：审批是 *能力推导出来的*，不是工具自己声明的——这意味着加新工具只要标对 capability，整套审批/沙箱/并行规则会**自动生效**。这是 trait-based 架构的复利。

## 6. 几个"标志性"工具的设计

### 6.1 `ExecShellTool`：背景化的 shell

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\shell.rs` 内部：
- 用 `SharedShellManager` 管理*所有*正在跑的命令；
- 默认走 `tokio::process::Command`，但如果 `ToolContext.sandbox_backend` 非空，**改走沙箱后端**（Linux Landlock / macOS Seatbelt / Bubblewrap / Docker）；
- 长跑命令进 *job* 模式，立即 emit `ToolResult::success("shell job started: {id}")`，**模型可以再调 `exec_wait` 等结果**；
- `ShellInteractTool` 可以往运行中的 PTY 发字符——支持 `top`、`vim` 这种交互式程序。

学习要点：**不是所有 shell 命令都有"立刻完成"的语义**。把"立刻完成"和"长跑"用 *同名工具的不同模式* 处理（参数 `wait: bool`），比拆成两个工具更直观。

### 6.2 `EditFileTool`：编辑后挂 LSP

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\file.rs` 的 edit 路径：
1. `resolve_path` 防越界；
2. 读旧文件 → string match → 写新内容；
3. 如果 `ToolContext.lsp_manager` 非空，调 `lsp_manager.collect_diagnostics_for(path).await`，把诊断结果**塞进 `ToolResult.metadata`**；
4. 引擎在 `flush_pending_lsp_diagnostics` 阶段把这些 diagnostics 转成 `<diagnostics>` block 拼到下一条 user 消息上。

> **学习要点**：把"编辑 → 编译反馈"这条闭环做成 *自动注入*，让模型 ***不必主动去问***——这是 Agent UX 的关键。

### 6.3 `LargeOutputRouter`：超大输出怎么办

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\large_output_router.rs`。
当 `read_file` 读了一个 500KB 的 log，直接塞回模型会爆 context。Router 的处理：

1. 检查输出 token 估算 > 阈值；
2. 调一个**轻量 V4-Flash 子代理**做 summary；
3. 把摘要返回给主模型，原始内容存到 `WorkshopVariables`；
4. 主模型如果想要完整内容，用 `promote_to_context` 工具把变量提取出来。

这就把"读大文件"和"决定要看哪一段"分开——**模型不需要把整个文件吞进自己的脑袋才能决策**。

## 7. MCP：把外部工具适配成 ToolSpec

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\mcp.rs` 实现 MCP 客户端：

```
~/.deepseek/mcp.json
        │
        ▼
McpPool: 管理多个 McpClient（每个连一个 server）
        │
        ▼
启动 stdio JSON-RPC 子进程  /  连 HTTP server
        │
        ▼
list_tools()  →  Vec<McpTool>
        │
        ▼
McpToolAdapter: impl ToolSpec
```

每个 MCP 服务器声明的工具被包成一个 `McpToolAdapter`，名字带前缀 `mcp__<server>__<tool>`，无缝塞进同一个 `ToolRegistry`。模型完全察觉不出"这是远程工具"。

> **学习要点**：MCP 的精妙在于**把"工具" 解耦成可热插拔的服务**。同一个 ToolSpec trait 既能落地内置工具，又能落地外部 RPC，作者只需要一个适配器。这是 trait-based 设计可扩展性的体现。

## 8. 子代理：递归地复用同一套基础设施

子代理 = 一个独立的 LLM 会话 + 共享同一个 `SubAgentManager`。

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\subagent\mod.rs:573-840`：

```rust
pub struct SubAgentRuntime {
    pub client: DeepSeekClient,
    pub model: String,
    pub auto_model: bool,
    pub workspace: PathBuf,
    pub max_steps: u32,
    pub manager: SharedSubAgentManager,    // ←— 给孙子用
    pub depth: u32,
    pub cancel_token: CancellationToken,
    /* … */
}
```

`SubAgentRuntime` 是把"父代理的能力"打包成 *可序列化运行时*。子代理跑起来的时候：
1. 也建一个 `Engine`；
2. 也接一对 `mpsc::channel<Op>` / `<Event>`；
3. 也调同一个 `ToolRegistry`；
4. 但它自己的 Op 不来自 UI，而是来自 `agent_spawn` 工具喂进来的 prompt；
5. 它的 Event 不去 UI，而是被 `subagent_progress` 收集器抽水后通过 `tx_subagent_completion` 回灌父代理。

> **核心 insight**：子代理不是一段"嵌套的特殊代码路径"，而是**同一个 Engine 跑在不同 channel 上**。整个项目的核心抽象就这么一份，复用到所有地方。

### 8.1 子代理为什么是 *background* 的

`spawn_background` 立刻返回 `agent_id`，不等子代理跑完。父代理可以：
- 继续做别的事；
- 或者用 `agent_wait` / `agent_result` 主动取；
- 或者什么都不做——等到 turn 结束时 `rx_subagent_completion` 会把结果灌回去（04 节那一段）。

这是为了让父代理 *并行* 跑多个调研任务而不阻塞，对应 `AGENTS.md` 里"长会话生存指南"的"全部委派出去"原则。

### 8.2 防爆栈：`max_spawn_depth`

每个子代理有 `depth` 字段。`agent_spawn` 拒绝执行如果 `depth >= max_spawn_depth`（默认 3）。否则一个错误的递归会瞬间起 200 个子代理把 token 烧光。

> **学习要点**：递归创建资源时，**一定要有 hard cap**，不能只靠"模型不会犯傻"——模型一定会犯傻。

## 9. RLM：把 Python REPL 当成"协处理器"

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\rlm.rs` 是一个工具：

```rust
pub struct RlmTool {
    client: Option<DeepSeekClient>,
    /* ... */
}
```

它干的事：
1. 用户调 `rlm` 工具，传 `task`（要做什么）+ `file_path`（长输入）；
2. 引擎起一个 Python REPL（`crates/tui/src/repl/`），把 file 内容塞进 `PROMPT` 变量；
3. REPL 里跑一个**子 LLM**，可以写 Python 代码用以下 *REPL 内函数*：
   - `llm_query(prompt)` —— 跑一次 sub-LLM
   - `llm_query_batched([prompt1, prompt2, …])` —— 并发跑多次
   - `rlm_query(...)` —— 递归一层 RLM
4. 子 LLM 写完 `FINAL("...")` 来宣告结果；
5. RlmTool 把 `FINAL` 的内容当成 `ToolResult.content` 返回。

**为什么这么绕**？因为长输入（200KB+）放进父模型的 context 会爆 token。把它**留在 REPL 里**，让子 LLM 写"chunk + map + reduce" 的代码自行处理——父模型只看到最终答案。

> **学习要点**：这是把 LLM 当成 "**程序合成器**" 而不是 "**段落生成器**" 用。子 LLM 不直接输出答案，它输出 *一段处理代码*，由 REPL 执行——本质上是 prompt-as-program。

注意 `AGENTS.md` 里强调：`llm_query` 等是 **REPL 内的 Python 函数**，**不是** 模型可见的工具。所以你不会在 `to_api_tools()` 里看到它们。这种"内部工具" / "外部工具"的对偶很巧妙。

## 11. 你新增一个工具时的 6 步清单

假设你要加一个 `cargo_clippy` 工具：

1. **新文件** `tools/clippy.rs`，定义 `pub struct CargoClippyTool;`
2. **`impl ToolSpec for CargoClippyTool`**：
   - `name() = "cargo_clippy"`
   - `description() = "..."`
   - `input_schema() = json!({...})` —— 至少声明个 `package: Option<String>` 字段
   - `capabilities() = vec![ToolCapability::ExecutesCode]` —— 这会自动让审批默认 *Required*
   - `execute(input, ctx) = …` —— 用 `ctx.shell_manager` 跑 `cargo clippy`
3. **在 `registry.rs` 里加 `with_clippy_tool()`** builder 方法。
4. **在 `engine.rs::build_tool_registry`** 里把它加上（注意 Plan 模式不应该挂 `ExecutesCode` 类工具）。
5. **写测试**：用 `ToolContext::new(tempdir())` 起一个轻量 ctx，喂个最小 cargo crate 跑通。
6. **更新 prompts/base.md** —— 把工具的使用提示加进去（或者依赖默认的 description）。

整个改动**完全本地**，不需要改 engine、不需要改 UI。这就是 trait-based 设计的最大价值。
