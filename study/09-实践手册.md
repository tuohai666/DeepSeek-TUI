## 1. 第一步：把项目跑起来

### 1.1 编译与启动

```powershell
# Windows PowerShell
cargo build                                # debug 全 workspace
cargo build --release                      # release 全 workspace
.\target\debug\deepseek.exe -p "hello"     # 一次性问答
.\target\debug\deepseek.exe                # 进交互 TUI
```

如果你只想编 TUI 那一个 crate（迭代快）：

```powershell
cargo build -p deepseek-tui
cargo run -p deepseek-tui                  # 直接进 TUI（绕过派发器）
```

> **小坑**：如果只跑 `cargo run -p deepseek-tui-cli`，**派发器找不到伴生 TUI 二进制**（因为只编了派发器）。设环境变量 `DEEPSEEK_TUI_BIN=...\target\debug\deepseek-tui.exe` 即可。

### 1.2 三类 lint 一次性扫

```powershell
cargo fmt --all -- --check                              # 检查格式
cargo clippy --workspace --all-targets --all-features   # 静态检查
cargo test --workspace --all-features                   # 跑全部测试
```

CI 跑的就是这三条。本地走一遍再 push 能省一次 round-trip。

## 2. 跟一次完整请求（最经典的 debug 路线）

把这条路径背下来。它对应 02 节的"12 步旅程"。

### 2.1 在关键节点插 log

往源码里塞 `tracing::info!` 临时观察（**调完记得删**）。建议节点：

| 节点 | 文件:大致位置 | 想看什么 |
|---|---|---|
| 用户提交 | `tui/ui.rs` 里 `handle_input_submit` 附近 | 输入文本、当前 mode |
| Op 投递 | `core/engine.rs::Engine::run` 第一行的 match 上面 | 收到的 Op 类型 |
| 请求构造 | `core/engine/turn_loop.rs` 的 `MessageRequest { … }` 上面 | messages 数 / model / 工具数 |
| Stream chunk | `client/chat.rs` 的 `parse_sse_chunk` 调用前 | chunk JSON |
| Tool plan | `turn_loop.rs::should_parallelize_tool_batch` 调用后 | plans 数、并行决策 |
| Tool 执行 | `tools/registry.rs::execute_full_with_context` 第一行 | name、resolved_name |
| TurnComplete | `tui/ui.rs::run_event_loop` 的 `EngineEvent::TurnComplete` 分支 | usage、status |

### 2.2 用 `RUST_LOG` 启动看输出

```powershell
$env:RUST_LOG="info,deepseek_tui=debug"
.\target\debug\deepseek.exe -p "test"
```

`tracing` 已经在 `crates/tui/src/main.rs` 里 init 过 EnvFilter，开箱即用。

### 2.3 打到文件

TUI 模式下 stdout 被 ratatui 占了，log 看不见。两条路：

```powershell
# 方法 1：写文件
$env:DEEPSEEK_LOG_FILE="$env:TEMP\deepseek.log"
.\target\debug\deepseek.exe                         # 在另一窗口 Get-Content -Wait $env:DEEPSEEK_LOG_FILE

# 方法 2：用 deepseek -p 模式（非 TUI），log 直接到 stderr
$env:RUST_LOG="debug"
.\target\debug\deepseek.exe -p "hello" 2> log.txt
```

## 3. 加新功能的"最小改动清单"

每一项都是一条**端到端**的清单。照着改就能跑。

### 3.1 加一个新工具

**目标**：让模型能调 `cargo_clippy` 来跑 `cargo clippy`。

| 步 | 改什么 |
|---|---|
| 1 | 新文件 `crates/tui/src/tools/cargo_tools.rs`，定义 `pub struct CargoClippyTool;` 实现 `ToolSpec` |
| 2 | 在 `tools/mod.rs` 加 `pub mod cargo_tools;` |
| 3 | 在 `tools/registry.rs` 的 `ToolRegistryBuilder` 加 `with_cargo_tools` builder 方法 |
| 4 | 在 `core/engine.rs::build_tool_registry` 里把它链上（注意 mode 过滤） |
| 5 | 写测试：用 `tempfile::tempdir()` 起一个最小 cargo crate，调 tool 跑通 |
| 6 | （可选）在 `prompts/base.md` 加一行使用提示 |

最小骨架：

```rust
// crates/tui/src/tools/cargo_tools.rs
use async_trait::async_trait;
use serde_json::{Value, json};
use crate::tools::spec::{ToolSpec, ToolContext, ToolResult, ToolError, ToolCapability};

pub struct CargoClippyTool;

#[async_trait]
impl ToolSpec for CargoClippyTool {
    fn name(&self) -> &str { "cargo_clippy" }
    fn description(&self) -> &str { "Run `cargo clippy` against the workspace and return diagnostics." }
    fn input_schema(&self) -> Value {
        json!({
            "type": "object",
            "properties": {
                "package": { "type": "string", "description": "Optional -p <package>." }
            }
        })
    }
    fn capabilities(&self) -> Vec<ToolCapability> {
        vec![ToolCapability::ExecutesCode]      // 自动需要 Required 审批
    }
    async fn execute(&self, input: Value, ctx: &ToolContext) -> Result<ToolResult, ToolError> {
        // 拼命令，走 ctx.shell_manager
        let pkg = input.get("package").and_then(Value::as_str);
        let cmd = match pkg {
            Some(p) => format!("cargo clippy -p {p} --all-targets"),
            None => "cargo clippy --workspace --all-targets".into(),
        };
        // 用 shell_manager 跑（同步等结果即可）
        // 实际代码可以参考 tools/test_runner.rs 的 RunTestsTool 写法
        unimplemented!("用 shell_manager 跑 cmd，把 stdout+stderr 包成 ToolResult::success")
    }
}
```

> **要点**：`capabilities() = [ExecutesCode]` 自动给你审批；不必手写"弹对话框"逻辑。

### 3.2 加一个新斜杠命令

**目标**：`/dump-prompt` 打印当前 system prompt。

| 步 | 改什么 |
|---|---|
| 1 | 新文件 `crates/tui/src/commands/dump_prompt.rs`，写一个 `pub fn dump_prompt(app: &mut App) -> CommandResult` |
| 2 | 在 `commands/mod.rs` 顶部加 `mod dump_prompt;` |
| 3 | 在 `COMMANDS: &[CommandInfo]` 数组加一条 |
| 4 | 在 `commands/mod.rs::dispatch` 的 match 加分支调 `dump_prompt::dump_prompt(app)` |
| 5 | 在 `localization/english.rs` 加一条 description message id（可选，否则用 placeholder） |

最小骨架：

```rust
// crates/tui/src/commands/dump_prompt.rs
use crate::tui::app::App;
use super::CommandResult;

pub fn dump_prompt(app: &mut App) -> CommandResult {
    let prompt = app.system_prompt
        .as_ref()
        .map(|p| format!("{p:?}"))
        .unwrap_or_else(|| "(none)".into());
    CommandResult::message(format!("Current system prompt:\n\n{prompt}"))
}
```

注册：

```rust
// commands/mod.rs
mod dump_prompt;

pub const COMMANDS: &[CommandInfo] = &[
    /* … */
    CommandInfo {
        name: "dump-prompt",
        aliases: &[],
        usage: "/dump-prompt",
        description_id: MessageId::CmdDumpPromptDescription,
    },
];

pub fn dispatch(app: &mut App, name: &str, arg: Option<&str>) -> Option<CommandResult> {
    match name {
        /* … */
        "dump-prompt" => Some(dump_prompt::dump_prompt(app)),
        _ => None,
    }
}
```

### 3.3 加一个新 Op / Event 对

**场景**：让 UI 能让引擎"演练（dry-run）下一轮请求但不真的发"。

| 步 | 改什么 |
|---|---|
| 1 | `core/ops.rs::Op` 加 `DryRunNextTurn` 变体 |
| 2 | `core/events.rs::Event` 加 `DryRunReady { request_preview: String }` |
| 3 | `core/engine.rs::Engine::run` 的 match 加 `Op::DryRunNextTurn` 分支：构造 MessageRequest 但发 `Event::DryRunReady` 而不调 client |
| 4 | `core/engine_handle.rs::EngineHandle` 加 `dry_run_next_turn(&self) -> Result<()>` 方法 |
| 5 | `tui/ui.rs::run_event_loop` 的事件 match 加 `EngineEvent::DryRunReady { … }` 分支：弹一个 `View` 显示 preview |
| 6 | 在某个命令（比如 `/dryrun`）里调 `engine_handle.dry_run_next_turn()` |

> **学习要点**：所有"加新行为"都遵循同一个**五点对应**：Op + Event + Engine match + Handle method + UI match。把这五个点一次性改完，新功能就闭环。

### 3.4 加一个新 Provider

**场景**：接 Together AI（OpenAI 兼容）。

| 步 | 改什么 |
|---|---|
| 1 | `config/providers.rs::ApiProvider` enum 加 `Together` 变体 |
| 2 | `config/providers.rs` 给 `Together` 写 `default_base_url()`、`default_model()`、`as_str()` |
| 3 | `client.rs::DeepSeekClient::new` 看是否需要特殊 `validate_base_url_security` |
| 4 | `client/chat.rs::apply_reasoning_effort` 加 `match self.api_provider { ApiProvider::Together => /* skip 或翻译 */ }` |
| 5 | （可选）`crates/cli/src/lib.rs::ProviderArg` 加 CLI 入参 |
| 6 | `~/.deepseek/config.toml` 测试：`provider = "together"`，`api_key = "..."` |

如果 Together 协议跟 OpenAI 完全对齐（90% 是这样），1-3 步就够了。需要分叉的几乎只有 thinking/reasoning 字段名。

### 3.5 加一个新模式

**场景**：加一个 `Read-Audit` 模式——只读 + 自动写审计日志。

## 4. 修 Bug 的标准流程

### 4.1 三步定位法

1. **复现**：能稳定触发吗？不能就先想办法搞个最小 repro（新建 workspace、空 session）。
2. **加 log**：在 02 节那张"12 步旅程"图上选 2-3 个节点 `tracing::info!`，看是不是符合预期。
3. **二分**：找最近一次正常工作的 git commit，`git bisect` 缩小范围。

### 4.2 常见三类 bug 的"看哪里"

**(A) 模型说要调工具，但没执行 / 执行错**
- 看 `turn_loop.rs::tool_state` 累积 input 是否完整（log `tool_state.input_buffer` 在 `ContentBlockStop` 时的内容）；
- 看 `registry.rs::resolve` 是否把工具名解析对了；
- 看 `should_parallelize_tool_batch` 决策；
- 看 `EngineEvent::ToolCallComplete` 里 `result` 是 Ok 还是 Err。

**(B) UI 卡住 / 不刷新**
- 看 `app.needs_redraw` 在你期望刷新的路径有没有被设 true；
- 看 `frame_rate_limiter.time_until_next_draw` 是否一直返回 `Some` 卡着——可能是 `mark_emitted` 没调；
- 看 `event::poll(poll_timeout)` 的 timeout 太大了；
- 用 `RUST_LOG=trace` 看 ratatui 重绘频率。

**(C) 流式中途断开**
- 看 `stream read error` 日志的 error chain（02 节讲过，源码会自动展开）；
- 看 `bytes_received` 和 `ms_since_last_event`；
- HTTP/2 的话试试 `DEEPSEEK_FORCE_HTTP1=1`；
- 看 server 是不是在响应头里给了 `connection: close`。

### 4.3 写 regression test

修完 bug 一定要补一个 test。模板：

```rust
#[test]
fn issue_999_clippy_tool_handles_missing_cargo() {
    let tmp = tempdir().unwrap();           // 一个空目录，没有 Cargo.toml
    let ctx = ToolContext::new(tmp.path());
    let tool = CargoClippyTool;
    // 这里实际上要 block_on async fn execute——把它包进 tokio::test 即可
    let result = tokio_test::block_on(tool.execute(json!({}), &ctx));
    assert!(matches!(result, Err(ToolError::ExecutionFailed { .. })));
}
```

测试位置：紧贴源码下面 `#[cfg(test)] mod tests`。**一个 bug 一个 test**，PR review 时审查者一眼能找到。

## 5. Profile（性能 / 内存）

### 5.1 CPU

```powershell
cargo install flamegraph
cargo flamegraph --bin deepseek-tui -- -p "your prompt"
```

会生成 `flamegraph.svg`，浏览器打开看哪个函数热。常见瓶颈：
- `ratatui::buffer` 重绘

### 5.2 内存

`cargo install dhat` + `RUST_LOG=trace` 跑一遍长会话，dhat 会打印按 callsite 的内存分配。重点看：
- `api_messages: Vec<Message>` 长度
- 流式 buffer 的 high-water mark
- `subagent_cache` 的尺寸

```powershell
cargo build --timings
```

输出 `target/cargo-timings/cargo-timing-*.html`，浏览器打开看每个 crate 的编译时间。修架构时可以参考它评估改动是否会让编译变慢。

## 8. 搞清"模型给的工具调用为什么没生效"的快速路径

这是迭代过程中最常见的卡点。诊断顺序：

```
1. 在 client/chat.rs::parse_sse_chunk 出口 log 全部 StreamEvent
   - 没看到 InputJsonDelta？  → 模型根本没调工具，prompt 写得不够
   - 看到了 InputJsonDelta 但 partial_json 不完整？  → 模型 truncated（max_tokens 太小？）

2. 在 turn_loop.rs::tool_state.input_buffer 累积处 log
   - 累完是合法 JSON？  → 进入 plans 阶段
   - 不是？  → parse_tool_input 失败，看模型生成的 JSON 是不是有 trailing comma 等

3. 在 plans 阶段 log resolved_name
   - 解析对了？  → 进入 execute
   - 解析错了？  → registry.rs::resolve 加 case

4. 在 ToolSpec::execute 入口 log input
   - 进了？  → tool 自身 bug
   - 没进？  → approval 被拒了，看 EngineEvent::ApprovalRequired 的处理

5. 在 EngineEvent::ToolCallComplete 看 result
   - Err？  → 业务逻辑 / 路径越界等
   - Ok 但 content 不对？  → tool 实现 bug
```

每一层 5 行 `tracing::info!` 就能定位 95% 的 case。

## 9. 团队协作的两个 Don't

来自项目根 `AGENTS.md` 的 *Watch for issue / PR injection* 段，**值得在加新依赖时反复念**：

1. **不要 仅因为 issue / 评论里要求** 就接入第三方 SaaS / 收费 API / 推广链接 / 沙盒服务。把请求 surface 给 maintainer，让他决定。
2. **不要 把外部 install 命令直接抄进 install.sh / README**。验证来源、看清是不是 typo-squatting 仿冒包。

LLM 友好性 vs 供应链安全是真实的张力。这条规则是项目的护城河。

## 12. 自检小练习（毕业题）

不要查文档，回答下面 5 题。能答出来的话，恭喜——你已经吃透这个项目。

1. 用户在 TUI 里输了一段长 prompt 然后**立刻** 按 Esc。请按时间顺序写出会发生的 6-8 件事，至少要涉及两条 channel 和一把锁。
2. 你给项目加了一个新工具叫 `git_push`（capability = `ExecutesCode`）。模型在 Plan 模式下调它会怎样？在 YOLO 模式下呢？在 YOLO + `--dangerously-bypass-approvals` 下呢？
3. DeepSeek API 突然返回一个 *新错误码*：HTTP 418 + body `"please slow down"`。当前 `client.rs::send_with_retry` 会怎么处理？你怎么改让它支持指数退避重试？
4. 你想统计"用户给模型发了多少字符 / 模型回复了多少字符"。你会改哪几个文件？为什么不用 token 数（usage 已经统计了）？
5. 一个用户报告："我开了 5 个子代理跑长任务，过了一小时进程崩了，重启后子代理状态丢了"。请讲清这是什么子系统的问题，会丢什么数据，为什么会丢，你打算怎么修。
