# 02 · 架构与数据流

> 这一节用一次"用户敲下回车"的旅程，把所有组件、channel、锁都串起来。读完之后，你应该能闭眼画出 *进程内部的拓扑图*。

## 1. 进程内部的拓扑图（先记住这一张）

```
┌──────────────── deepseek 进程（派发器）─────────────────┐
│  clap 解析 args                                         │
│  ConfigStore::load → ResolvedRuntimeOptions             │
│  Command::new("deepseek-tui").env(…).args(…).status()   │
└────────────┬─────────────────────────────────────────────┘
             │ fork (Child Process)
             ▼
┌──────────────── deepseek-tui 进程（主进程）──────────────┐
│  ┌──────────────┐                  ┌──────────────────┐  │
│  │   TUI Task   │ ◀──── Event ───── │   Engine Task    │  │
│  │  (ratatui)   │ ─────  Op ─────▶ │ (mpsc receiver)   │  │
│  │              │                   │                   │  │
│  │  键盘 + 鼠标 │                   │   ┌────────────┐  │  │
│  │  + UI 渲染   │                   │   │  Tools     │  │  │
│  └──────┬───────┘                   │   │ Registry   │  │  │
│         │                           │   └─────┬──────┘  │  │
│         │ tokio::spawn              │         │         │  │
│         ▼                           │   ┌─────▼──────┐  │  │
│   ┌──────────────┐                  │   │ DeepSeek   │  │  │
│   │ TaskManager  │                  │   │ Client     │  │  │
│   │ (background) │                  │   │ (HTTP/SSE) │  │  │
│   └──────────────┘                  │   └────────────┘  │  │
│                                     │                   │  │
│                                     │   ┌────────────┐  │  │
│                                     │   │ SubAgent   │  │  │
│                                     │   │ Runtime    │  │  │
│                                     │   └────────────┘  │  │
│                                     │                   │  │
│                                     │  Persistence ──▶  │  │
│                                     │  Actor (~/.deepseek)│  │
│                                     └──────────────────┘  │
└────────────────────────────────────────────────────────┘
```

**记住三点**：
1. **两个进程**——派发器只活几毫秒，TUI 进程活整个会话期。
2. **TUI 进程内部两个主 Task**——UI Task 和 Engine Task，靠 `mpsc::channel` 通话。
3. **Engine 是异步的瑞士军刀**——它持有 `DeepSeekClient`、`ToolRegistry`、`SubAgentManager`、`TaskManager`、`McpPool`、`LspManager` 等大约十个组件。

## 2. 一次完整请求的旅程（按时间线）

假设你输了 `帮我把 turn_loop.rs 里的 stream_started_at 重命名为 stream_start_time`，按下回车。下面 12 步发生在 1–10 秒内：

### 步骤 1：UI 收到回车键

`run_event_loop` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tui\ui.rs:641`) 在 `tokio::select!` 里同时听键盘和引擎事件。回车键触发 `handle_input_submit`，最终把用户消息封成 `Op::SendMessage`（`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\ops.rs:14-33`）：

```rust
Op::SendMessage {
    content: "帮我…".to_string(),
    mode: AppMode::Agent,
    model: "deepseek-v4-pro",
    reasoning_effort: Some("high"),
    auto_model: false,
    allow_shell: true,
    trust_mode: false,
    auto_approve: false,
    approval_mode: ApprovalMode::Suggest,
    /* … */
}
```

### 步骤 2：Op 通过 channel 投递到引擎

`engine_handle.tx_op.send(op).await`，进 `mpsc::channel(32)`（容量 32）。引擎那边的 `Engine::run` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs:569`) 是一个 `while let Some(op) = self.rx_op.recv().await` 循环。

### 步骤 3：引擎把消息塞进 session

`handle_send_message` 把 `content` 包成 `Message{role:"user", content: vec![ContentBlock::Text{...}]}` 推到 `session.messages`，同时更新 `working_set`、刷新系统 prompt（合并项目上下文 / 用户 memory / 当前 mode 的指令）。

### 步骤 4：构建 ToolRegistry

引擎根据 `mode`、`features`、`runtime_services` 通过 `ToolRegistryBuilder` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\registry.rs:393-820`) 拼出能给模型用的工具集合。Plan 模式只给只读工具，Agent 模式给完整工具集，YOLO 模式额外解掉审批门。

### 步骤 5：组装 MessageRequest

```rust
// @D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:239-260
let request = MessageRequest {
    model: self.session.model.clone(),
    messages: self.messages_with_turn_metadata(),
    max_tokens: effective_max_output_tokens(&self.session.model),
    system: self.session.system_prompt.clone(),
    tools: active_tools.clone(),
    tool_choice: if active_tools.is_some() { /* auto/required */ } else { None },
    reasoning_effort: effective_reasoning_effort,
    stream: Some(true),
    /* … */
};
```

### 步骤 6：发起 SSE 流

`client.create_message_stream(request).await` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\client.rs:736-741`) → 内部走 `handle_chat_completion_stream` → 用 `reqwest` 发 POST，返回一个 `Stream<Item = Result<StreamEvent>>`。每个 `data:` 行被解析成一个 `StreamEvent::ContentBlockStart` / `Delta` / `Stop` 等。

### 步骤 7：引擎逐 chunk 处理流

```rust
// 简化版，详见 @D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:336-684
while let Some(event_result) = stream.next().await {
    match event_result? {
        StreamEvent::ContentBlockDelta { delta: Delta::TextDelta { text }, .. } => {
            tx_event.send(Event::MessageDelta { content: text, .. }).await?;
        }
        StreamEvent::ContentBlockDelta { delta: Delta::InputJsonDelta { partial_json }, .. } => {
            tool_state.input_buffer.push_str(&partial_json);
            // 累积工具调用 JSON，stop 时再 parse
        }
        StreamEvent::ContentBlockStop { /* … */ } => {
            /* 工具调用就绪 → 发 Event::ToolCallStarted */
        }
        /* … */
    }
}
```

每个 `Event::MessageDelta` 立即推到 UI Task。

### 步骤 8：UI 把 delta 写到屏幕

UI 端 `match event` 那一段 (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tui\ui.rs:701-720`)：

```rust
EngineEvent::MessageDelta { content, .. } => {
    let sanitized = sanitize_stream_chunk(&content);
    current_streaming_text.push_str(&sanitized);
    let index = ensure_streaming_assistant_history_cell(app);
    app.streaming_state.push_content(0, &sanitized);
    let committed = app.streaming_state.commit_text(0);
    if !committed.is_empty() {
        append_streaming_text(app, index, &committed);
    }
}
```

下一帧 `Frame Rate Limiter` 触发 `terminal.draw(…)`，新字就出现在屏幕上。

### 步骤 9：模型决定要调 `edit_file`

模型 stream 出一个 `tool_use` block：
```json
{ "name": "edit_file", "input": { "path": "...", "old_str": "...", "new_str": "..." } }
```

引擎攒齐这个 block 后，进入 **plans 阶段**：根据工具能力判断是否需要审批、是否能并行、是否只读。

### 步骤 10：审批闸门（如果需要）

`edit_file` 是 `WritesFiles` 类——`approval_required = true`。引擎发 `Event::ApprovalRequired`，UI 弹出对话框；用户按 `y` → UI 调 `engine_handle.approve_tool_call(id)` → 引擎从 `rx_approval` 收到 `Approved`。

### 步骤 11：执行工具

`registry.execute_full_with_context("edit_file", input, Some(context))` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tools\registry.rs:124-187`)：

1. 找到 `EditFileTool`；
2. 执行 → 返回 `ToolResult { content: diff, metadata: …}`;
3. 大输出经 `LargeOutputRouter` 路由（>阈值 token 时走 workshop 替换为摘要）；
4. 引擎挂 LSP post-edit hook 收 diagnostics；
5. 引擎把结果包成 `Message{role:"user", content: vec![ContentBlock::ToolResult{...}]}` 加入 session；
6. 发 `Event::ToolCallComplete`。

### 步骤 12：再喂回模型，直到没有 tool_use

`turn_loop` 的外层 `loop`：只要这一步还产生了 `tool_uses`，就再发一个新请求（带上更新后的 messages），直到模型只回文本不调工具。这就是"step"的意义——一个 turn 内可能有 N 个 step。

最后发 `Event::TurnComplete{usage, status: Completed}`，UI 把 spinner 关掉，写 session checkpoint。

## 3. 数据流分类（控制 vs 内容 vs 持久化）

**三类数据沿着不同通道流动**。混淆它们就读不懂源码：

### 3.1 控制流（Op / Event）

```
UI ──┐
     │ mpsc::Sender<Op>  (cap=32)
     ▼
   Engine
     │ mpsc::Sender<Event>  (cap=256)
     ▼
    UI
```

**特点**：
- `Op` 容量小（32）——UI 偶尔发，且操作是用户主动行为。
- `Event` 容量大（256）——流式 chunk 极快，缓冲必须够。
- 还有几条**专用控制 channel**：
  - `tx_approval: mpsc::Sender<ApprovalDecision>` ——审批专用
  - `tx_user_input: mpsc::Sender<UserInputDecision>` ——`request_user_input` 工具专用
  - `tx_steer: mpsc::Sender<String>` ——turn 进行中时插话
  - `cancel_token: Arc<StdMutex<CancellationToken>>` ——共享的取消信号
  - `tx_subagent_completion: mpsc::UnboundedSender<SubAgentCompletion>` ——子代理完成事件回灌

来源：`Engine::new` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs:402-565`)。

> **学习提示**：每多一类专用 channel，是因为发生过"它和别的事件混用导致漏处理"的 bug。挨个读 channel 名，能感受到这是个被实战打磨过的设计。

### 3.2 内容流（HTTP/SSE 与磁盘）

引擎与外部世界的内容沿这些通道：

| 方向 | 通道 | 文件 |
|---|---|---|
| Engine → DeepSeek API | HTTP/2 SSE | `crates/tui/src/client/chat.rs` |
| Engine → 本地 shell | `tokio::process::Command` | `tools/shell.rs` |
| Engine → MCP 服务器 | stdio JSON-RPC 或 HTTP | `mcp.rs` |
| Engine → LSP 服务器 | stdio JSON-RPC | `lsp/client.rs` |
| Engine → 子代理 | 进程内 `tokio::spawn` 的 task | `tools/subagent/mod.rs` |
| Engine ↔ 磁盘（session） | Persistence Actor | `tui/persistence_actor.rs` |
| Engine ↔ 磁盘（snapshot） | side-git repo | `snapshot/` |

### 3.3 持久化流（写穿 actor）

为什么要单独一个 **Persistence Actor**？因为 fsync 是阻塞的，UI 不能等。看 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tui\ui.rs:417-420`：

```rust
if let Ok(persist_manager) = SessionManager::default_location() {
    let handle = persistence_actor::spawn_persistence_actor(persist_manager);
    persistence_actor::init_actor(handle);
}
```

UI 端用 `persistence_actor::persist(PersistRequest::Checkpoint(session))` 投递请求，**立即返回**；actor 在专用 task 里串行写文件。

这是经典的 **Actor 模式**：把可能阻塞的副作用搬到独立 task，主流程只发消息。

## 4. 几把关键的"锁"和它们防什么

`Engine` 持有的同步原语清单：

```rust
// @D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs:299-346
pub struct Engine {
    /* … */
    cancel_token: CancellationToken,
    shared_cancel_token: Arc<StdMutex<CancellationToken>>,
    tool_exec_lock: Arc<RwLock<()>>,
    capacity_controller: CapacityController,
    /* … */
}
```

| 同步原语 | 防什么 |
|---|---|
| `CancellationToken` | 用户按 Esc 时让一切并发任务（流、工具、子代理）尽快收手 |
| `Arc<StdMutex<CancellationToken>>` | UI 持有一个引用便于在它的同步代码里调 `.cancel()` |
| `tool_exec_lock: Arc<RwLock<()>>` | 写锁让"会改文件的工具"串行执行；读锁让只读工具并行 |
| `subagent_manager: Arc<RwLock<SubAgentManager>>` | 多个 task 同时增减子代理表 |
| `mcp_pool: Arc<AsyncMutex<McpPool>>` | MCP 服务器连接池，独占访问 |

> **学习提示**：注意 `tool_exec_lock` 的 `RwLock<()>` 用法——它根本不存任何数据，只是一个 *并行许可证*。读锁就是"我是只读工具，并行无所谓"，写锁就是"我要写盘，别人都让开"。这是一个非常巧妙的并发模式。

## 5. 一张"线程级"时序图

把上面的 12 步配上线程标注：

```
时间 ─▶
[UI Task]
  键盘事件 ─▶ Op::SendMessage ──┐
  ┊                             │
  ┊                             ▼
[Engine Task]                  收 Op
                                │
                                ├─▶ session.add_message(user)
                                ├─▶ ToolRegistryBuilder.build()
                                ├─▶ client.create_message_stream() ──┐
                                                                      │
                                                                      ▼
                                                          [Stream Future]
                                                          每收到 SSE chunk
                                                                      │
                                       Event::MessageDelta ◀──────────┘
                                                │
[UI Task]                                      ▼
  收 Event ─▶ append to history cell ─▶ next frame draws
                                                ┊
                                                ┊  (流结束)
                                                │
[Engine Task]                                   ▼
                                        plans ⇒ outcomes
                                                │
                                                ├─[parallel branch]─▶ FuturesUnordered
                                                │                       │
                                                │                       ├─▶ tool A
                                                │                       └─▶ tool B
                                                │
                                                ├─[serial branch]─▶ for plan in plans
                                                │                     │
                                                │                     ├─▶ 弹审批
                                                │                     └─▶ tool.execute
                                                ▼
                                        Event::ToolCallComplete
                                                │
                                                ▼
                                        loop continues (next step)
                                                ┊
                                                │ (no more tool_uses)
                                                ▼
                                        Event::TurnComplete
                                                │
[UI Task]                                       ▼
  收 TurnComplete ─▶ flush active cell ─▶ 写 session snapshot
```

注意 **同一个 Engine Task 既驱动 stream 又驱动工具执行**——没有专门的"工具线程"。这是为什么 `cancel_token` 必须共享：取消必须能同时打断 stream future 和正在跑的工具 future。
