# 04 · 引擎与轮次循环：项目的心脏

> **如果只读一节，读这一节**。
> 这里的代码量大、状态多、并发凶，但抓住三层结构（**`Engine::run` → `handle_send_message` → `handle_deepseek_turn`**），其余都是这三层的展开。
> 配合源码同步看，强烈建议把 `crates/tui/src/core/engine.rs` 和 `crates/tui/src/core/engine/turn_loop.rs` 同时打开。

## 1. 三层心跳

```
Engine::run                    ← 进程级心跳：永远在 recv Op
  └─ handle_send_message       ← 用户级心跳：一次 SendMessage = 一个 Turn
       └─ handle_deepseek_turn ← API 级心跳：一个 Turn 内 N 次模型调用 + 工具
            └─ stream chunk loop ← 字符级心跳：流式 SSE 解码
```

每一层职责清晰：
- 外层只关心"什么时候停"；
- 中层关心"这一轮我要做什么准备/收尾"；
- 内层关心"模型说了什么、要不要调工具"；
- 最里层关心"流式数据的字节解码"。

## 3. 第一层：`Engine::run` —— 进程级心跳

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs:569-799`，简化结构：

```rust
pub async fn run(mut self) {
    while let Some(op) = self.rx_op.recv().await {
        match op {
            Op::SendMessage { content, mode, model, /*…*/ } => {
                self.handle_send_message(content, mode, model, /*…*/).await;
            }
            Op::CancelRequest => {
                self.cancel_token.cancel();
                self.reset_cancel_token();
            }
            Op::SpawnSubAgent { prompt } => { /* … */ }
            Op::ListSubAgents => { /* 列举并 emit AgentList */ }
            Op::SetCompaction { config } => { /* 改阈值 */ }
            Op::SyncSession { messages, … } => { /* 恢复会话 */ }
            Op::CompactContext => { self.handle_manual_compaction().await; }
            Op::Rlm { content, model, child_model, max_depth } => {
                self.handle_rlm(content, model, child_model, max_depth).await;
            }
            Op::EditLastTurn { new_message } => { /* 撤回最后一轮再发 */ }
            Op::Shutdown => { break; }
            /* … */
        }
    }
}
```

**这就是引擎的全部主循环**——一个 `while let Some(op) = self.rx_op.recv().await`。每个 Op 串行处理。

> 留意：**没有"等模型流"那段代码出现在 run 里**。它在更深一层的 `handle_send_message → handle_deepseek_turn` 里，那里会用 `tokio::select!` 处理流。运行期，绝大多数时间这个 `while` 在 `handle_send_message` 里阻塞，外层的 `rx_op.recv()` 看不到新消息。

### 3.1 取消是怎么工作的

`Op::CancelRequest` 的两行代码隐藏一个精彩的设计：

```rust
self.cancel_token.cancel();       // 当前 token 触发
self.reset_cancel_token();         // 立刻换一把新 token
```

为什么要 reset？因为 `CancellationToken` 是 *一次性* 的——一旦 `cancel()`，所有挂在它上面的 future 立刻收到信号，但**这个 token 本身永远是 cancelled 状态**。如果不换新的，下一次 SendMessage 时 stream 一开始就被取消了。

`reset_cancel_token` (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs:351-362`)：

```rust
fn reset_cancel_token(&mut self) {
    let token = CancellationToken::new();
    self.cancel_token = token.clone();
    match self.shared_cancel_token.lock() {
        Ok(mut shared) => { *shared = token; }
        Err(poisoned) => { *poisoned.into_inner() = token; }
    }
}
```

`shared_cancel_token` 是给 UI 层的 `EngineHandle.cancel()` 用的——UI 不需要 await，它在键盘事件回调里同步调 `.cancel()`。所以这个 token 必须用 `Arc<StdMutex<…>>` 包起来，让 UI 同步代码也能读到 *最新* 的那一把。

> **学习要点**：`CancellationToken` 不能反复 cancel/reset，所以你看到的所有"重置"都是**整把扔掉换新的**。这是 tokio_util 给的契约。

## 4. 第二层：`handle_send_message` —— 一轮的准备/收尾

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs` 中（约 800-1500 行），这一层做以下事情：

## 5. 第三层：`handle_deepseek_turn` —— **本节的灵魂**

### 5.1 骨架（伪代码）

```rust
pub(super) async fn handle_deepseek_turn(&mut self, turn: &mut TurnContext, /*…*/) -> (Status, Option<String>) {
    let client = self.deepseek_client.clone().unwrap();
    let mut tool_catalog = tools.unwrap_or_default();
    let mut active_tool_names = initial_active_tools(&tool_catalog);
    let mut loop_guard = LoopGuard::default();
    let mut stream_retry_attempts = 0u32;

    loop {                                                       // ← step 循环
        if self.cancel_token.is_cancelled() { return Interrupted; }

        // 1. 处理 steer 输入
        while let Ok(steer) = self.rx_steer.try_recv() { /* 加到 messages */ }

        // 2. 自动压缩（如果阈值到了）
        if should_compact(...) { compact_messages_safe(...).await; }

        // 3. capacity guardrail 检查点
        if self.run_capacity_pre_request_checkpoint(...).await { continue; }

        // 4. token 预算检查（>100k 时进 recovery）
        if estimated_input > input_budget { recover_context_overflow(...).await; }

        // 5. 把累积的 LSP diagnostics 注入下一条 user message
        self.flush_pending_lsp_diagnostics().await;

        // 6. 拼请求
        let request = MessageRequest { /*…*/ };

        // 7. 起流
        let mut stream = client.create_message_stream(request.clone()).await?;

        loop {                                                   // ← chunk 循环
            tokio::select! {
                _ = self.cancel_token.cancelled() => break,
                result = timeout(chunk_timeout, stream.next()) => match result {
                    Ok(Some(Ok(event))) => process_stream_event(event),
                    Ok(Some(Err(e))) => maybe_transparent_retry(e),
                    Ok(None) => break,                  // 流正常结束
                    Err(_) => break_on_chunk_timeout(),
                }
            }
        }

        // 8. 流结束。决定怎么走

        if stream_died_with_nothing && stream_retry_attempts < 3 {
            stream_retry_attempts += 1;
            continue;                                             // 静默重试
        }

        // 9. 把这一步的 assistant message 推进 session
        self.add_session_message(Message { role: "assistant", content: blocks }).await;

        // 10. 没有工具调用 → 看有没有 sub-agent 完成事件 / inline REPL
        if tool_uses.is_empty() {
            if let Some(completions) = drain_subagent_completions { /* 注入 */ continue; }
            if has_repl_block { /* 跑 paper-spec RLM */ continue; }
            return Completed;                                      // 真的结束
        }

        // 11. 有工具调用 → 进入 plans/outcomes 两阶段调度
        let plans = build_tool_plans(tool_uses, /*…*/);
        let outcomes = if parallel_allowed {
            execute_in_parallel(plans).await
        } else {
            execute_sequentially(plans).await
        };

        // 12. 把 tool result 推进 session，再循环
        for outcome in outcomes { add_tool_result_message(outcome); }
        turn.next_step();
    }
}
```

把这 12 个步骤背下来，整本源码都好读了。

## 6. chunk 循环里的精妙处

### 6.1 SSE 事件处理（步骤 7）

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:469-683`。事件分四类：

```rust
match event {
    StreamEvent::MessageStart { message } => { usage = message.usage; }

    StreamEvent::ContentBlockStart { index, content_block } => {
        // content_block 可能是 Text / Thinking / ToolUse / ServerToolUse
        // 给 UI 发 MessageStarted / ThinkingStarted；ToolUse 不立刻发，等 stop
    }

    StreamEvent::ContentBlockDelta { index, delta } => {
        match delta {
            Delta::TextDelta { text } => { /* push 给 UI */ }
            Delta::ThinkingDelta { thinking } => { /* push 给 UI */ }
            Delta::InputJsonDelta { partial_json } => {
                // 工具参数是分片到达的 JSON 字符串
                tool_state.input_buffer.push_str(&partial_json);
                if let Some(value) = parse_tool_input(&tool_state.input_buffer) {
                    tool_state.input = value;
                }
            }
        }
    }

    StreamEvent::ContentBlockStop { index } => {
        // 关键：只有这里才 emit Event::ToolCallStarted
        // 否则 UI 会先看到 input={}, 然后才看到 input={...}, 闪烁
        if matches!(stopped_kind, Some(ContentBlockKind::ToolUse)) {
            tx_event.send(Event::ToolCallStarted {
                id: tool_state.id.clone(),
                name: tool_state.name.clone(),
                input: final_tool_input(tool_state),
            }).await;
        }
    }

    StreamEvent::MessageDelta { usage } => { /* 累计 usage */ }
    StreamEvent::MessageStop | StreamEvent::Ping => {}
}
```

> **学习要点 1**：`Delta::InputJsonDelta` 是分片到达的 *字符串*。这意味着工具参数 JSON **可能在中途无法被 parse**——`{"path": "src/` 这种半句。代码用 `parse_tool_input` 探测式 parse，能 parse 就更新，不能 parse 就继续累积。
> **学习要点 2**：`ToolCallStarted` 必须在 `ContentBlockStop` 才发，**因为 input 必须最终化**。否则 UI 卡的就是半句 JSON，渲染出来很丑。

### 6.2 透明重试（步骤 7→8）

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:411-466`。

如果流刚开始连接就断了（用户**还没看到任何字**），代码会**静默重试**，不上报错误：

```rust
if should_transparently_retry_stream(any_content_received, transparent_stream_retries, …) {
    transparent_stream_retries += 1;
    drop(stream);
    match client.create_message_stream(stream_request.clone()).await {
        Ok(fresh) => {
            stream = fresh;
            stream_start = Instant::now();
            continue;
        }
        Err(retry_err) => { /* 报错并 break */ }
    }
}
```

判断条件 `any_content_received`：只要收到过任何 content block delta，就**不再静默重试**——因为用户已经看到字了，悄悄重新发会让 UI 重复显示。

> **学习要点**：透明重试是用户体验的细节。代码用 `any_content_received: bool` 这个一行的状态变量把"是否可以悄悄重来"和"必须报错"分开。这是把"用户视角"翻译成"程序状态"的好例子。

### 6.3 thinking_to_persist 的兜底

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:739-750`：

## 7. 工具调度的两阶段：plans 与 outcomes

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:1042-1500+`。

### 7.1 阶段一：planning（同步）

```rust
let mut plans: Vec<ToolExecutionPlan> = Vec::with_capacity(tool_uses.len());
for (index, tool) in tool_uses.iter_mut().enumerate() {
    /* 1. 解析工具名（含 hallucinated 名字 fuzzy resolve） */
    /* 2. 检查 caller_allowed（root vs subagent 的可见性） */
    /* 3. 找 spec → 决定 approval_required / supports_parallel / read_only */
    /* 4. LoopGuard 检查（同一个工具同样参数被连续调，告警/拦截） */
    plans.push(ToolExecutionPlan { /* … */ });
}
```

`ToolExecutionPlan` 是个**纯数据**结构——拿到所有工具的 metadata 之后，引擎可以**全局** 决定怎么调度。

### 7.2 阶段二：outcomes（异步）

```rust
let parallel_allowed = should_parallelize_tool_batch(&plans);
let outcomes = if parallel_allowed && plans.len() > 1 {
    /* FuturesUnordered，全部并发 */
} else {
    /* for plan in plans，串行 */
};
```

`should_parallelize_tool_batch` 的规则：**全部** plan 都满足 `read_only && supports_parallel && !approval_required` 才能并行。任何一个写工具就降级到串行。这就是 `tool_exec_lock: Arc<RwLock<()>>` 设计的目的——读类是 `read().await`，写类必须 `write().await`，**自然形成读写分离调度**。

> **学习要点**：这是经典的 **gather/scatter** 模式。先把 N 个工具的 metadata 统一收集成 plans，然后根据 metadata 整体决定是 fan-out 并发还是串行。这种"先决策再执行"的两阶段写法在并发系统里很常见。

### 7.3 为什么并行度这么保守？

因为：
1. 写工具并发会导致 *file race*（`apply_patch` 改同一文件）；
2. 审批一次只能弹一个对话框，并行没法弹；
3. shell 工具会争抢 PTY；
4. 模型并不是真的在"独立判断"——它是按顺序写出 tool_call 的，自己脑子里有顺序假设。

所以默认走串行，并行只是"系统知道这一批确实彼此独立"时的优化。

## 8. 子代理完成的"挂起恢复"

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\turn_loop.rs:818-920`。

正常情况：模型不调工具就 `break`，turn 结束。
**特殊情况**：模型还有子代理在跑（`agent_spawn` 后异步），这时不能直接结束——必须等子代理完成。

```rust
if tool_uses.is_empty() {
    let mut completions = Vec::new();
    while let Ok(c) = self.rx_subagent_completion.try_recv() { completions.push(c); }
    if completions.is_empty() {
        let running = { let mgr = self.subagent_manager.read().await; mgr.running_count() };
        if running > 0 {
            tx_event.send(Event::status(format!("Waiting on {running} sub-agent(s)"))).await;
            tokio::select! {
                biased;
                _ = self.cancel_token.cancelled() => return Interrupted,
                Some(c) = self.rx_subagent_completion.recv() => completions.push(c),
                Some(steer) = self.rx_steer.recv() => { /* 用户插话 */ }
            }
        }
    }
    if !completions.is_empty() {
        for c in completions {
            self.add_session_message(subagent_completion_runtime_message(&c.payload)).await;
        }
        turn.next_step();
        continue;                              // 喂回模型
    }
    /* 真的结束 */
    break;
}
```

**这是"等待 + 接力"模式的教科书写法**：
1. 先 try_recv 已经攒好的；
2. 没有就 await 下一个；
3. 同时给用户保留 steer 通道（防止饿死）。

> **学习要点**：`tokio::select!` 的 `biased` 关键字让分支按声明顺序检查——这里把 cancel 放第一，确保用户能优先打断。

## 9. 系统 prompt 的"动态拼装"

不是每次请求都重新拼。源码用一个哈希避免抖动：

```rust
// @D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine.rs:447-449
let stable_prompt = Some(system_prompt);
session.last_system_prompt_hash = Some(system_prompt_hash(stable_prompt.as_ref()));
session.system_prompt = stable_prompt;
```

每轮 `refresh_system_prompt(mode)` 重新拼一遍，但只有哈希变了才 emit `SessionUpdated`。**这是为了让 DeepSeek 的 prefix cache 命中——如果系统 prompt 字节级别一致，前缀缓存就给你 90% 折扣**。

`prompts/base.md` 的开头几句也是按这个原则——稳定的 instruction 放前面，会变的（如 `## Environment` 里的 `cwd`、`lang`）放后面。

> **学习要点**：在大模型应用里，**字节稳定性 = 真金白银**。你看到代码里好多 `OnceLock`、`sort_by(|a,b| a.name().cmp(b.name()))` 都是为了让序列化结果**逐字节一致**，让 KV cache 能复用。

## 10. 容量守门员：`capacity_controller`

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\engine\capacity_flow.rs`。

每次请求前：
1. 估算 input tokens (`estimated_input_tokens`)；
2. 跑 `CapacityController::observe`，给出风险带（`Healthy` / `Watch` / `Warn` / `Critical`）；
3. 风险高时执行 `GuardrailAction`：
   - `RequestCompaction` → 触发自动压缩
   - `ReplayWithSmallerWindow` → 切到只用最近 N 轮重发
   - `BlockUntilSafe` → 拒绝发请求，让用户 `/compact` 或 `/clear`

这一段的精彩之处是它**不是一锤子**——而是带"内存"的：每次 turn 结束把"我做了什么决策、用户/模型的反应是什么"写进 `capacity_records.jsonl`，下次启动时回放。这是一种轻量的**自适应控制**，不需要训练模型，靠规则统计就能用。

学到了：**做防护要"轻量持久"**——不要每次进程重启都"重新学一次"。

## 11. 对照表：你以后修引擎时该改哪儿

| 你想改的事 | 改哪个文件 |
|---|---|
| 增加新 Op（如 `/teleport`） | `core/ops.rs` + `Engine::run` 的 match + `EngineHandle` 加方法 |
| 增加新 Event（如 `Event::ToolDryRun`） | `core/events.rs` + 触发处 + UI 端 match |
| 改并发策略（如允许 grep 类并行） | `turn_loop.rs::should_parallelize_tool_batch` |
| 改流式重试次数 | `turn_loop.rs` 顶部的 `MAX_STREAM_RETRIES` 常量 |
| 改 token 预算 | `core/engine.rs::context_input_budget` |
| 改 capacity 阈值 | `core/capacity.rs::CapacityControllerConfig` |
| 改自动压缩触发条件 | `compaction.rs::should_compact` |
| 加 capacity 自适应记录字段 | `core/capacity_memory.rs::CapacityMemoryRecord` |
