# 08 · 会话与上下文管理：让对话能跑一整天

> 一次对话 30 分钟可能没事；跑一天会撞上四件大事：**token 爆炸、进程崩溃、工作区被改坏、上下文里全是死信息**。
> 这一节看作者用哪四套机制对付这四件事。

---

## 1. 四件大事 × 四套机制

| 大事 | 机制 | 文件 |
|---|---|---|
| **进程崩溃** 丢失对话 | Session 持久化 + checkpoint | `session_manager.rs`, `tui/persistence_actor.rs` |
| **工作区被改坏** 想回滚 | Side-git Snapshot | `snapshot/` |
| **上下文 token 爆炸** | Compaction（手动 + 自动 + capacity 守门） | `compaction.rs`, `core/capacity.rs` |
| **跑太久"上下文都是死信息"** | Cycle 切片 + Briefing | `cycle_manager.rs` |

## 2. Session：磁盘上的对话状态

### 2.1 数据结构

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\session_manager.rs:101-117`：

```rust
pub struct SavedSession {
    pub schema_version: u32,
    pub metadata: SessionMetadata,                // id / title / model / workspace / token / mode
    pub messages: Vec<Message>,                   // 完整对话
    pub system_prompt: Option<String>,            // 当时的 system prompt
    pub context_references: Vec<SessionContextReference>,  // @path / @file 提及
}
```

存储位置：`~/.deepseek/sessions/<id>.json`，一个 session 一个文件。

### 2.2 写入路径：原子写

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\session_manager.rs:158-170`：

```rust
pub fn save_session(&self, session: &SavedSession) -> std::io::Result<PathBuf> {
    let path = self.validated_session_path(&session.metadata.id)?;
    let content = serde_json::to_string_pretty(session)?;
    write_atomic(&path, content.as_bytes())?;     // ← NamedTempFile + fsync + rename
    self.cleanup_old_sessions()?;                 // ← 超过 50 个删旧的
    Ok(path)
}
```

**为什么 `write_atomic`** 而不是 `fs::write`？因为崩溃时 `fs::write` 可能写到一半——你打开文件看到一段截断的 JSON。`write_atomic` 用 `NamedTempFile + persist`：
1. 先写到 `<id>.json.tmp`；
2. fsync 到磁盘；
3. `rename` 覆盖到目标——POSIX 保证原子。

> **学习要点**：**任何关键状态文件都要走原子写**。Linux 的 ext4 在崩溃时能给你一个空文件——除非用 rename。

### 2.3 路径校验：防注入

`validated_session_path` 不允许 `id` 含 `/` `\` `..`，只允许 `[A-Za-z0-9_-]`。否则模型一句"saved as foo/../../etc/passwd" 就把别人文件覆盖了——这是工程上的常见硬约束。

### 2.4 读出路径：前缀模糊

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\session_manager.rs:289-300+`：

```rust
pub fn load_session_by_prefix(&self, prefix: &str) -> std::io::Result<SavedSession> {
    let sessions = self.list_sessions()?;
    let matches: Vec<_> = sessions.iter().filter(|s| s.id.starts_with(prefix)).collect();
    if matches.len() > 1 { /* 报错 */ }
    self.load_session(&matches[0].id)
}
```

`deepseek resume abc12` 不必输完 32 字符 UUID。这种小细节是 CLI 工具的可用性。

### 2.5 Checkpoint：随机崩溃也能救回

`session_manager.rs::save_checkpoint`：
- 路径：`~/.deepseek/sessions/checkpoints/<id>.checkpoint.json`
- 时机：**每收到一个 `EngineEvent::SessionUpdated` 都写一次**（流式中也写）
- 用途：进程崩溃时，下次 `deepseek resume <id>` 优先从 checkpoint 恢复
- 清理：`Event::TurnComplete` 触发 `PersistRequest::ClearCheckpoint`

代码里 (`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tui\ui.rs:1098-1103`)：

```rust
if (app.is_loading || app.is_compacting)
    && let Ok(manager) = SessionManager::default_location()
{
    let session = build_session_snapshot(app, &manager);
    persistence_actor::persist(PersistRequest::Checkpoint(session));
}
```

> **学习要点**：常态用 session.json，崩溃用 checkpoint.json，**正常退出后清掉 checkpoint**——避免下次启动覆盖正常 session。

### 2.6 Persistence Actor：把磁盘 IO 搬到独立 task

来自 `@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\tui\persistence_actor.rs`：

```
init_actor(handle)         // 进程内全局 OnceCell 持有 actor handle
persist(req)               // 任意线程发请求，立即返回（unbounded channel）
                ▼
            actor task
                ▼
        match req:
          SessionSnapshot(s) → save_session
          Checkpoint(s)      → save_checkpoint
          ClearCheckpoint    → clear_checkpoint
          Shutdown           → break
```

UI 线程 / 引擎线程 *永不* `tokio::fs::write(...).await`。永远是 `persistence_actor::persist(req)`，立即返回。

## 3. Snapshot：side-git 工作区快照

### 3.1 为什么要"side-git"

**问题**：用户的 `.git` 是他自己的版本控制。如果 deepseek 直接往里面 `git commit`，会污染用户的 commit 历史。
**方案**：在 `~/.deepseek/snapshots/<workspace-hash>/` 下另起一个 *bare 风格* git 仓，把工作区文件 add+commit 进去。每次 turn 结束创建一个 commit，commit message 里包含 turn id。`/restore` 命令把 commit checkout 回工作区。

**关键不变量**：`.git` 永远不被碰。用户做 `git status` 跟 deepseek 完全无关。

### 3.2 触发点

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\turn.rs::post_turn_snapshot`，在每次 `Event::TurnComplete` 后调一次。如果 turn 修改了文件就提交一个 commit；没改就跳过。

### 3.3 实现细节（高层）

`crates/tui/src/snapshot/` 下的代码用 `gix`（pure-Rust git）而不是 `git` 子进程，避免依赖。它做：
1. 维护一份独立的 working dir 元数据；
2. 用 `cp -R`-级别的复制把工作区文件同步过去；
3. 调 `gix` API 做 `add . && commit -m "turn N"`；
4. `/restore N` 从快照 checkout。

看代码时注意它会**避开 `.git`、`.deepseek`、`target/` 等**，避免把巨型构建产物快照进去——这有性能和磁盘双重价值。

> **学习要点**：当你的工具需要给用户的工作区做"安全网"时，*绝对不要* 篡改用户的 VCS。另起一个 side-store。

## 4. Compaction：让上下文不爆

### 4.1 配置

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\compaction.rs:29-36`：

```rust
pub struct CompactionConfig {
    pub enabled: bool,
    pub token_threshold: usize,      // 超过这个 token 数就触发
    pub model: String,               // 用哪个模型做摘要（一般是 Flash）
    /* … */
}
```

DeepSeek V4 是 1M 上下文，但官方建议 **>200K** 后效果开始下降。`token_threshold` 默认 ~150K——给后面留余量。

### 4.2 触发器：`should_compact`

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\compaction.rs:613-...`：

```rust
pub fn should_compact(messages: &[Message], config: &CompactionConfig, workspace: Option<&Path>) -> bool {
    if !config.enabled { return false; }
    let estimated = estimate_message_tokens(messages, workspace);
    estimated > config.token_threshold
}
```

调用点在 `turn_loop.rs` 里——每次 step 开始检查一次，大了就压缩。

### 4.3 `compact_messages_safe`

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\compaction.rs:867-1000+`。流程：

1. **Pin 区段**：最近 N 条消息原样保留（`pinned_recent_messages` 函数），让模型有"我刚才说了什么"的上下文；
2. **Summary 区段**：把更早的消息丢给压缩模型（一般 Flash），让它生成结构化摘要：
   - 用户提了什么问题
   - 完成/未完成的工具调用
   - 关键文件路径
   - 决定但还没执行的事
3. **拼回**：用一个 `system` 消息装 summary，加上 pinned 区段，构成新的 messages 数组；
4. **替换 session.messages**；
5. **emit `EngineEvent::CompactionCompleted`** —— UI 顶部显示"Compacted N → M messages"。

`_safe` 后缀的承诺：**绝不损坏原始 messages**——压缩失败就保留原数组重试 next turn。

> **学习要点**：压缩不是 *去掉* 消息，而是 *换一个紧凑的同义表示*。结构化摘要比"丢掉前 50%"更优秀，因为它保留了"决定/状态"，丢的只是"过程"。

### 4.4 Capacity Controller：硬防御

光靠"超过阈值才压缩"不够——压缩本身要花 30 秒，期间用户可能又发了新消息。Capacity 是另一道闸：

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\core\capacity.rs`，在每次请求前 *主动* 评估：

```
estimated_input_tokens
       │
       ▼
CapacityController::observe(metrics) → DecisionBand
       │
       ├── Healthy   → 直接发
       ├── Watch     → 发，但同时排队 compaction
       ├── Warn      → 拦截，先 compaction，完成后再发
       └── Critical  → 拒绝发，弹"会话太大，请 /compact 或 /clear"
```

Capacity 的精妙之处：它**有持久化记忆**。`~/.deepseek/capacity_records.jsonl` 记录每一轮的 token 增长 / 风险带 / 是否拦截 / 用户反应。
下次启动时回放，模型不需要重新学校准——这是**轻量自适应**。

## 5. Cycle Manager：长任务的"翻篇"

### 5.1 问题

跑了 5 小时，session 里有 800 条 message。即使 compaction 也只能压到 200K token，但其中 90% 都是早期废弃路径——模型还得读一遍才能找到现在的目标。

### 5.2 方案：**Cycle**

把会话切片成"**1 个 cycle = 1 个聚焦阶段**"。当一段任务收尾时（手动 `/cycle next` 或自动触发），引擎：
1. 让模型写一份 **briefing**（这一阶段做了什么、剩下什么、关键路径）；
2. 把所有当前 messages 归档到 cycle history；
3. 用 *只有 briefing* 重启 session.messages —— 上下文从 200K 降到 5K；
4. emit `EngineEvent::CycleAdvanced { from, to, briefing }` 给 UI。

代码：`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\cycle_manager.rs`。

### 5.3 `CycleBriefing` 数据

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\cycle_manager.rs:144-...`：

```rust
pub struct CycleBriefing {
    pub cycle: u32,
    pub created_at: chrono::DateTime<chrono::Utc>,
    pub model: String,
    pub summary: String,
    pub objectives_done: Vec<String>,
    pub objectives_pending: Vec<String>,
    pub key_paths: Vec<String>,
    pub open_questions: Vec<String>,
    pub token_estimate: usize,
}
```

每个 briefing 是个**结构化文档**而不是大段散文。下一 cycle 启动时，briefing 通过模板渲染回 system prompt，模型立刻知道"我接着干什么"。

### 5.4 Cycle vs Compaction

两者都是"把上下文变小"，但语义不同：

| | Compaction | Cycle |
|---|---|---|
| 触发 | 自动（token 阈值） | 手动 / 自动（任务完成） |
| 粒度 | 渐进（保留 pinned） | 重置（只留 briefing） |
| 何时用 | 一个任务还没做完，但 token 太多 | 一个任务做完，开始下一个 |
| 结果 | 同会话 + 压缩 | 同会话 + 切片，可回看历史 cycle |

> **学习要点**：长任务的关键不是"上下文窗口够大"，而是"**只把当下需要的东西放在上下文里**"。Cycle 让用户能 *主动声明* "切换主题"，从根本上避免上下文里堆满旧目标。

## 7. 离线队列：网络挂了也别丢消息

`@D:\1os\DeepSeek-TUI-tuohai666\crates\tui\src\session_manager.rs:36-68`：

```rust
pub struct OfflineQueueState {
    pub schema_version: u32,
    pub session_id: Option<String>,
    pub messages: Vec<QueuedSessionMessage>,
    pub draft: Option<QueuedSessionMessage>,
}
```

场景：
1. 用户在 turn 跑的时候连续发了 3 条新消息（流式过程中按回车）；
2. 模型还没结束，UI 把消息排进 `app.queued_messages`；
3. 同时 `persistence_actor::persist(PersistRequest::QueueSnapshot)` 写到 `~/.deepseek/sessions/queue.json`；
4. 进程崩溃；
5. 下次 `deepseek resume` 自动检测同一 session_id 的队列文件，恢复 `app.queued_messages`，UI 顶部提示"恢复了 3 条排队消息"。

这是**对用户输入的尊重**——你已经按了回车，无论何种崩溃都不该让你重打。

## 8. 会话生命周期总览

```
deepseek 启动
    │
    ├── 不带 resume → 创建 SavedSession (id=UUID, messages=空, system_prompt=default)
    │
    └── deepseek resume <id>
        ├── load_session(id) → SavedSession
        ├── 同时检查 checkpoint → 如果存在 + 比 session 新 → 用 checkpoint
        ├── 同时检查 OfflineQueue → 如果 session_id 匹配 → 恢复
        └── 喂给 App + Engine

进入 run_event_loop
    │
    ├── 用户每发一条消息 → SyncSession Op → Engine::session.messages.push
    ├── 每 chunk 流式更新 → SessionUpdated Event → checkpoint 写
    ├── Turn 结束 → SessionSnapshot → 主 session.json 写 + checkpoint clear
    ├── 阈值到达 → 自动 compaction
    ├── 用户 /cycle next → cycle 切片
    └── 用户 /restore N → snapshot 回滚

退出
    │
    └── persistence_actor Shutdown → 把队列里所有还没写的 flush 完再退
```

整套机制保证：**任何崩溃 / 网络中断 / 关机重启，最多丢失最近一个 chunk 的进度**。

## 9. 它们之间怎么不打架

四套机制都是"对 messages 的修改"，作者用以下规则避免冲突：

1. **Compaction / Cycle 后 emit `EngineEvent::SessionUpdated`** —— UI 同步刷新自己的 `app.api_messages`；
2. **Snapshot 不改 messages，只改 workspace 文件** —— 与 compaction 正交；
3. **Persistence actor 是单 task 串行写** —— 不会出现 session.json 和 checkpoint.json 并发写撕开；
4. **Capacity 不直接动 messages**，只发 `RequestCompaction` action 让 Engine 主流程处理。

> **学习要点**：多套机制如果都改同一份状态，最危险。要么把它们 *都通过同一个 actor*（这里是 Engine），要么给它们划清"各自的状态"。
